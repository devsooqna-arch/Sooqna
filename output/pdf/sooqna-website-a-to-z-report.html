1. ملخص تنفيذي

موقع سوقنا هو منصة إعلانات عربية باتجاه RTL شبيهة بفكرة eBay Kleinanzeigen. الهدف الواضح من المنتج هو تمكين المستخدم من البحث عن إعلانات محلية حسب التصنيف والمدينة، إضافة إعلان جديد، التواصل مع البائع، حفظ المفضلة، وإدارة الحساب والإعلانات.

المشروع ليس مجرد واجهة ثابتة. يوجد Frontend مستقل بـ Next.js وBackend مستقل بـ Express/Prisma/PostgreSQL، مع Firebase Authentication للمصادقة فقط، والبيانات التجارية موجودة في قاعدة البيانات عبر REST API.

الحالة العامة

  • الصفحة الرئيسية والإعلانات تعمل محلياً وتعرض بيانات حقيقية من API.
  • المصادقة مبنية على Firebase Auth، والـ backend يتحقق من ID token.
  • إضافة الإعلانات محمية بتسجيل الدخول وتأكيد البريد.
  • تمت إضافة حماية ضد تكرار نشر الإعلان عبر clientRequestId.
  • يوجد CI/build/test scripts، وworkflow نشر عبر PM2/SSH.

7. Authentication Details

الموقع يستخدم Firebase Auth بشكل مباشر من الواجهة لتسجيل الدخول، إنشاء الحساب، Google sign-in، إرسال بريد التحقق، واستعادة كلمة المرور. الـ backend لا يرسل SMTP لهذه الوظائف؛ دوره هو التحقق من token وتطبيق policies.

التدفق

  • المستخدم يسجل عبر Firebase Client SDK.
  • الواجهة تحصل على ID token.
  • apiClient يرسل Authorization: Bearer token للطلبات المحمية.
  • Backend middleware يتحقق عبر Firebase Admin.
  • المستخدم غير مؤكد البريد يتم منعه من الإعلانات والرسائل والمفضلة.

Email Verification / Reset

  • إرسال رابط تحقق البريد يتم عبر Firebase.
  • استعادة كلمة المرور تتم عبر Firebase.
  • يوجد logging آمن لمحاولات الإرسال بدون كشف بيانات حساسة.
  • يوجد cooldown 60 ثانية لإعادة إرسال التحقق.
  • أخطاء Firebase مثل auth/too-many-requests تظهر برسالة عربية أوضح.

8. Listings Flow

إنشاء إعلان

  • الواجهة تستخدم wizard من 4 خطوات.
  • التحقق من الحقول يتم قبل الانتقال والنشر.
  • الصور تُرفع بعد إنشاء draft.
  • بعد الصور يتم publish.
  • يمكن تفعيل featured listing.

منع التكرار

تمت إضافة clientRequestId من الواجهة ويتم حفظه على مستوى الإعلان. يوجد unique index على ownerId + clientRequestId. إذا وصل نفس الطلب مرتين، يرجع backend الإعلان الموجود بدل إنشاء إعلان جديد.

Lifecycle

draftpublishedarchivedsoldrejected
  • النشر يتطلب صوراً.
  • الإعلان المنشور له تاريخ انتهاء حسب إعدادات البيئة.
  • يدعم renew وexpire وunpublish وfeature.

9. API & Database

API Modules

  • /api/auth: session, resend verification, recaptcha verify.
  • /api/users: profile/me.
  • /api/listings: CRUD/lifecycle/images.
  • /api/favorites: add/remove/list favorites.
  • /api/messages: conversations/messages/read/unread.
  • /api/categories: categories directory.
  • /api/uploads: listing/profile images.
  • /api/reports, /api/audit, /api/engagement, /api/contact.

Database Models

النماذج الأساسية: User, Listing, ListingImage, Favorite, Conversation, ConversationParticipant, Message, Category, Upload, Report, AuditLog, EngagementEvent.

Validation

الـ backend يستخدم Zod schemas للتحقق من body/query/params قبل دخول controller. هذا يقلل الأخطاء ويحافظ على contracts واضحة.

10. Testing, CI/CD, Deployment

Testing

  • Backend: Jest + ts-jest.
  • Frontend: TypeScript lightweight tests عبر tsx.
  • Playwright e2e موجود في web.
  • اختبارات مضافة حديثاً لتأكيد البريد، reset email، resend cooldown، listing idempotency.

Build

  • Web build: npm run build يشمل redirects validation, Next build, CSS assertion, branding assertion.
  • Backend build: tsc -p tsconfig.json.
  • Backend typecheck منفصل.

Deployment

  • GitHub Actions CI موجود.
  • Deploy workflow يعتمد على SSH وPM2.
  • Backend وWeb على نفس السيرفر مع reverse proxy.
  • متغيرات البيئة مطلوبة لكل من Firebase, DB, CORS, uploads.
ملاحظة: آخر محاولة push فشلت بسبب صلاحيات GitHub للحساب الحالي. الكومِت موجود محلياً لكن يحتاج حساب لديه صلاحية كتابة على الريبو.

11. Risks & Recommendations

مخاطر/نقاط انتباه

  • يجب التأكد من Firebase Authorized Domains في الإنتاج.
  • يجب تشغيل Prisma migrations في الإنتاج قبل نشر الكود الجديد.
  • صفحات dev يجب ألا تظهر في production إن لم تكن محمية جيداً.
  • يوجد warnings قديمة في EditListingPageView.tsx تحتاج تنظيف.
  • يجب التأكد من سياسة تخزين وحذف الصور المرفوعة.

أولويات مقترحة

  1. تحسين guest/protected state في صفحة نشر إعلان.
  2. إكمال تعريب جميع الرسائل التقنية.
  3. إضافة tests تفاعلية للـ listing wizard.
  4. مراجعة responsive mobile بشكل تفصيلي.
  5. إضافة monitoring/logging production للأخطاء المهمة.
